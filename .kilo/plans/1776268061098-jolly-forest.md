# Plan: Fix Harvard Architecture Mismatch for printf()

## Diagnosis: Why is UART empty?

The lack of UART output is caused by a **Harvard Architecture memory aliasing mismatch** in the CPU's design.

1. **The Issue:** Your C program's string literals (like `"\r\n=================================\r\n"`) are stored in the read-only `.rodata` section, which the linker script (`link.ld`) correctly places in the `ROM` block starting at `0x80000000`. 
2. **The Hardware Limitation:** In your current `riscv_core.v` design, the datapath's memory read (`alu_result`) is ONLY routed to `data_memory` (at `0x00000000`) and the MMIO peripherals (`timer`, `gpio`). The `instruction_memory` module is completely disconnected from the datapath's read/write interface.
3. **The Result:** When `printf("%s")` executes an `LBU` instruction to load characters from the string at `0x8000XXXX`, the hardware fails to route this address to `instruction_memory`. Instead, it aliases to `data_memory.v`, reading random uninitialized zeros (null characters `\0`). Because the string instantly evaluates to `NULL`, `printf` finishes immediately without outputting any characters to the UART.

## Proposed Implementation Plan

To make the CPU capable of executing standard C programs with read-only data, we must implement **Unified Memory Access** and standardize the bootloader.

### Step 1: Hardware Fix - Dual-Port Instruction Memory
We will modify `instruction_memory.v` to act as a True Dual-Port ROM.
*   **Port A** will remain dedicated to the `PC` for instruction fetching.
*   **Port B** will be added to accept `data_addr` and `funct3` from the CPU datapath. We will implement the exact same byte/half/word extraction logic (handling `LBU`/`LB`/`LW`) currently found in `data_memory.v`.

### Step 2: Hardware Fix - Core Routing
We will update `riscv_core.v` to:
*   Connect the ALU address and `funct3` to the new Port B on `instruction_memory`.
*   Update the `mem_read_data` multiplexer. If `alu_result` is in the `0x8000_XXXX` range (ROM access), it will route the read value from the instruction memory instead of the data memory.

### Step 3: Software Fix - Bare-Metal C Bootloader
Currently, global variables initialized to 0 (like `tick_count`) happen to work because FPGA block RAM defaults to 0. However, fully compliant C code requires explicit memory initialization.
*   Update `sw/link.ld` to expose memory boundary symbols (`__data_load_start`, `__data_start`, `__data_end`, `__bss_start`, `__bss_end`).
*   Update `sw/start.S` to include a standard assembly copy-loop (to copy `.data` variables from ROM to RAM) and a clear-loop (to zero out `.bss` variables) before jumping to `main()`.

By executing this plan, your CPU will fully support standard C compilation behavior and `printf` will successfully access string literals from the ROM.
