# Plan: Port and Run FreeRTOS on Custom RISC-V CPU

## Diagnosis & Feasibility

The CPU is remarkably close to being fully FreeRTOS-ready. The official FreeRTOS RISC-V port has two primary hardware requirements:
1. **Synchronous Context Switching:** Handled via the `ecall` instruction, which triggers a trap. Our `riscv_core.v` supports this and correctly sets `mcause = 11`.
2. **Tick Generation:** Handled via a Machine Timer Interrupt. Our custom `timer.v` implements `mtime` and `mtimecmp` registers exactly as the FreeRTOS port expects (we just need to point FreeRTOS to our `0xFFFF0000` base address), and `csr_file.v` correctly maps this to `mcause = 7`.

**Current Limitations to Address:**
- **RAM Size:** Currently, Data Memory is only 8KB. FreeRTOS requires RAM for Task Control Blocks (TCBs), task stacks, and its own heap. 8KB is extremely tight. We need to expand this to 32KB.
- **Trap Vector Setup:** The hardware initializes the `mtvec` CSR to `0x00000000`, and `start.S` never updates it. We must point `mtvec` to the FreeRTOS trap handler so the kernel can intercept `ecall` and timer interrupts.
- **Build System:** The `sw/` folder lacks a Makefile to compile multiple C files (FreeRTOS core files + `main.c`).

## Proposed Implementation Plan

### Step 0: Version Control & Worktree Setup
- Stage and commit all current changes (the Harvard architecture fix) to the main branch.
- Create a new Git worktree (e.g., in `../simple_riscv_cpu_freertos` or simply a new branch `feature/freertos` depending on specific tool constraints, but aiming for a worktree as requested) to isolate the FreeRTOS implementation from the stable bare-metal base.

### Step 1: Hardware RAM Expansion (8KB -> 32KB)
- **`rtl/data_memory.v`**: Increase the memory arrays from `0:2047` to `0:8191`.
- **`sw/link.ld`**: Update the RAM `LENGTH` from `8K` to `32K`.
- **`sw/start.S`**: Update the initial stack pointer (`sp`) address to `0x00008000` (top of 32KB RAM).

### Step 2: Fetch & Integrate FreeRTOS
- Clone the official `FreeRTOS-Kernel` into `sw/FreeRTOS` (via `git clone --depth 1`).
- Provide a standard `FreeRTOSConfig.h` tailored for our CPU:
  - Define CPU clock rate (e.g., 50MHz, depending on FPGA/Simulation).
  - `#define configMTIME_BASE_ADDRESS 0xFFFF0000`
  - `#define configMTIMECMP_BASE_ADDRESS 0xFFFF0008`
  - Define heap size (e.g., `16 * 1024` bytes).
- Create a dummy `freertos_risc_v_chip_specific_extensions.h` (required by the GCC RISC-V port).

### Step 3: Software Trap Setup
- **`sw/start.S`**: Add assembly instructions to load the address of `freertos_risc_v_trap_handler` into the `mtvec` CSR right before jumping to `main`. This allows FreeRTOS to take control of all exceptions and interrupts.

### Step 4: Build System & Application
- **`sw/Makefile`**: Create a Makefile to compile `main.c` alongside the essential FreeRTOS files (`tasks.c`, `queue.c`, `list.c`, `timers.c`, `heap_4.c`, `port.c`, `portASM.S`) using `riscv64-unknown-elf-gcc`. It will output `main.elf`, extract `.bin`, and use `bin_to_mem.py` to generate the `.mem` file for Verilog.
- **`sw/main.c`**: Replace the current bare-metal loop with a FreeRTOS application. We will create two tasks (e.g., "Task 1" and "Task 2") that alternate printing to the UART using `vTaskDelay`, demonstrating successful context switching and timer ticks.
