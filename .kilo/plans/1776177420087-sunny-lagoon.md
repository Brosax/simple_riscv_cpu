# 修复计划：Vivado 2019.1 `data_memory` RAM 推断失败

## 现象与结论（基于你提供的 synthesis log）
当前问题已基本定位，不在 `instruction_memory`，而在 `data_memory`：

- `instruction_memory` 已被正常识别：
  - 日志有 `Trying to map ROM "mem" into Block RAM`
  - 综合统计也显示 `instruction_memory` 下有 `ROMs := 1`
- `data_memory` 在后续优化阶段失败：
  - `[Synth 8-6849] Infeasible attribute ram_style = "block" ... trying to implement using LUTRAM`
  - `[Synth 8-2914] Unsupported RAM template`（定位到 `rtl/data_memory.v:16`）
  - `[Synth 8-5743] Unable to infer RAMs due to unsupported pattern`

这说明：Vivado 2019.1 对当前 `data_memory` 的写法不认为是可推断 BRAM 的模板，导致试图转成 LUTRAM，最终触发模板不支持错误。

## 根因分析（针对 Vivado 2019.1 推断规则）
`data_memory` 当前组合了以下高风险模式：

1. 同一块 `reg [31:0] mem[...]` 被两个时序块（CPU/JTAG）同时读写。
2. 以位片方式写入（`mem[word][7:0] <= ...` 等 byte-lane 部分赋值）。
3. 两端口都带字节写使能（`we_mask` + `debug_mem_wstrb`）。
4. 还叠加了调试端口读控制语义。

以上在新版本 Vivado 可能还能过，但在 2019.1 很容易超出“受支持 RAM 模板”。

## 推荐修复方案（主方案）
将 `data_memory` 重构为“4 个 8-bit RAM bank”结构（`mem_b0..mem_b3`），保持接口不变。

### 设计要点
- 存储体从：
  - `reg [31:0] mem [0:2047]`
- 改为：
  - `(* ram_style="block" *) reg [7:0] mem_b0 [0:2047];`
  - `(* ram_style="block" *) reg [7:0] mem_b1 [0:2047];`
  - `(* ram_style="block" *) reg [7:0] mem_b2 [0:2047];`
  - `(* ram_style="block" *) reg [7:0] mem_b3 [0:2047];`

### 端口行为
- CPU 端口：
  - 按 `we_mask[3:0]` 分别写 4 个 bank。
  - 同步读 4 个 bank 后拼接成 `raw_cpu_rdata`。
- Debug 端口：
  - 按 `debug_mem_wstrb[3:0]` 分别写 4 个 bank。
  - 同步读 4 个 bank 后拼接成 `debug_mem_rdata`。
- `funct3` 的 LB/LH/LW/LBU/LHU 扩展逻辑保持不变。

### 为什么这个方案更稳
- 避免了 `mem[word][31:24]` 这种“位切片写 RAM”的推断歧义。
- 每个 bank 都是标准 8-bit RAM 读写模板，更符合 Vivado 2019.1 识别习惯。
- 不改变 `riscv_core` 与上层接口，功能风险较小。

## 备选方案（当主方案仍失败时）
### 方案 B：限制 debug 端口能力（降复杂度保实现）
- 仅允许 debug 读，不允许 debug 写（或用仲裁与 CPU 复用单写口）。
- 优点：推断难度显著降低。
- 代价：JTAG 调试写内存能力受限。

### 方案 C：实例化 XPM（强约束）
- 使用 `xpm_memory_tdpram`（`ifdef SYNTHESIS`）+ 仿真行为模型。
- 优点：最可控。
- 代价：代码侵入更高、可移植性变差。

## 实施步骤
1. 重构 `rtl/data_memory.v` 为 4-bank 结构（不改模块端口）。
2. 保留原有 load/store 语义与调试语义。
3. 确认不存在同一 bank 在同一 always 中复杂条件混写模式。
4. 保留 `instruction_memory` 当前写法（它已在日志中推断成功）。

## 验证步骤
1. 单测优先：
   - `make _unit_data_memory`
2. 相关集成：
   - `make integration-tests`
3. Vivado 复测：
   - `vivado -mode batch -source basys3_top.tcl`
4. 通过标准：
   - 不再出现 `[Synth 8-2914] Unsupported RAM template`
   - 不再出现 `[Synth 8-5743] Unable to infer RAMs due to unsupported pattern`
   - `data_memory` 不再报 `[Synth 8-6849] Infeasible attribute ... trying to implement using LUTRAM`

## 非阻塞问题（后续再清理）
- `uart_tx.v` 存在状态编码位宽截断/不可达 case 警告（与当前 RAM 失败无直接因果）。
- 大量 unconnected debug 端口是告警，不是本次综合失败根因。

## 风险与回滚
- 风险：双端口同地址同周期写冲突语义需明确（CPU 与 debug 同时写同地址）。
- 控制：先保持与现有行为一致（“最后赋值优先”或工具默认），必要时增加冲突约束。
- 回滚：保留当前版本分支，若行为不一致可回退并转方案 B/C。

## 新问题排查：FPGA 上“流水灯跑一下就卡住”

### 结论（高置信度）
最可能不是 CPU 真正“卡死”，而是 `basys3_top` 里**看门狗逻辑主动把 LED 冻结**，表现为“跑一下后停住”。

关键代码在 `rtl/basys3_top.v`：
- `watchdog_counter` 在 `host_write_enable` 触发时清零。
- 若 `host_write_enable` 超过约 2 秒不再出现（`24_000_000` @ 12MHz），`watchdog_expired` 置 1。
- LED 旋转只在 `!watchdog_expired` 时执行；超时后设计就是“冻结灯”。

这意味着：
- 只要程序没有持续写 `tohost`（`0x80002000`），即使 CPU 正常运行，LED 也会被判定为“挂死”。
- 你当前 `test_hello.mem` 很可能只在前期写几次 `tohost`，随后进入延时/循环，触发该误判。

### 佐证
1. `riscv_core` 中 `host_write_enable = real_mem_write && (alu_result == TOHOST_ADDR)`，本质是“程序事件”，不是“CPU 心跳”。
2. `rtl/test_hello.mem` 指令序列是若干次 `sw` 到 `tohost` 后进入循环/收尾，不保证每 2 秒持续写 `tohost`。
3. 你反馈“能先跑一下再卡住”与该门限逻辑完全吻合（时间窗后冻结）。

### 次要风险点（低置信度）
- `instruction_memory` 使用了绝对路径 mem 文件；若后续 bitstream/工程迁移，可能加载错误程序，但这通常会导致“从一开始就异常”，不太像“先正常后冻结”。
- `uart_tx` 的状态机告警需要后续清理，但与 LED 冻结时序不直接匹配。

## 修复方案（按优先级）

### 方案 A（推荐，最小改动）
修改 `basys3_top` 的 LED 逻辑：**看门狗超时不再冻结流水灯**。
- 保留 `watchdog_expired` 标志，仅作为错误指示位（例如 `led[15]` 常亮）。
- 流水灯继续旋转，避免误以为系统停机。

适用场景：当前 LED 主要用于“活性展示”，不是严格 fault latch。

### 方案 B（更合理的活性判定）
把看门狗喂狗源从 `host_write_enable` 改为“CPU 前进证据”，例如：
- 连接 `debug_pc_value`（已存在端口），检测 PC 在一段时间内是否变化；
- 或检测 `instruction_done` 脉冲（需从 core 导出）。

优点：看门狗基于真实执行活性，不依赖程序是否写 `tohost`。

### 方案 C（短期验证）
临时增大门限（例如 2s -> 10s/30s）验证误判假设。
- 若“卡住时间”随门限线性变化，基本可确认是看门狗策略问题而非 CPU 死锁。

## 实施与验证计划
1. 先做方案 A：取消“超时冻结灯”，保留状态指示。
2. 重新综合、实现、烧录。
3. 观察：
   - 流水灯是否持续运行；
   - `watchdog_expired` 指示位是否在约 2 秒后置位（若仍使用 2 秒门限）。
4. 若需要真正“卡死检测”，再迭代到方案 B（PC/指令级心跳）。

## 新需求计划：编写 1~9 循环程序并显示到 Basys3 七段数码管

### 目标
- CPU 程序循环输出 `1,2,3,4,5,6,7,8,9,1,2...`
- FPGA 上七段数码管稳定显示当前数字。
- 与现有 `tohost` 机制兼容，尽量少改 core。

### 方案选择（已确定）
采用**最小侵入方案**：
1. 程序继续通过写 `TOHOST_ADDR (0x80002000)` 输出值（当前系统已有）。
2. `basys3_top` 捕获 `host_write_enable + host_data_out[3:0]`，作为七段显示数据源。
3. 在 top 中新增七段译码与动态扫描逻辑（默认只亮最右一位，其他位熄灭，减少复杂度）。

这样无需新增 MMIO 外设、无需改 `riscv_core` 地址译码，风险最低。

### 代码改动计划

1. **`rtl/basys3_top.v`**
   - 顶层端口新增：
     - `output wire [6:0] seg`
     - `output wire dp`
     - `output wire [3:0] an`
   - 新增寄存器：
     - `reg [3:0] sevenseg_digit;`（锁存最近一次输出数字）
   - 在 `host_write_enable` 有效时：
     - `sevenseg_digit <= host_data_out[3:0];`
   - 新增 7 段译码组合逻辑（Basys3 为常阳极，`seg/an/dp` 通常低有效）：
     - 支持 `1..9` 的段码
     - 其他值显示空白或 `0`
   - 位选逻辑：
     - `an = 4'b1110`（仅最低位亮）
     - `dp = 1'b1`（小数点灭）
   - 同时修复先前误判：看门狗不再冻结 LED（保留状态指示即可）。

2. **`constraints/basys3.xdc`**
   - 追加七段相关引脚约束（来自 `Basys3_XDC_example.txt`）：
     - `seg[0..6]`, `dp`, `an[0..3]`
   - 保持时钟/LED/UART 约束不变。

3. **`rtl/test_hello.mem`（或替换为新的程序 mem）**
   - 写入一个 RV32I 循环程序：
     - 初始化寄存器 `xN=1`
     - 循环：
       - 将当前值写到 `0x80002000`
       - 延时
       - `xN = xN + 1`
       - 若 `xN == 10` 则 `xN = 1`
       - 跳回循环
   - 保证 `host_data_out[3:0]` 始终是 `1..9`。

### 验证计划

1. **仿真级快速验证**
   - 运行：`make integration-tests`
   - 关注 `tohost` 写入序列是否按 `1..9` 变化。

2. **综合实现验证（你本地执行）**
   - 重新综合/实现/烧录。
   - 预期现象：
     - 七段最右位循环显示 `1..9`
     - LED 不再“跑一下卡住”（若按计划已去掉冻结行为）。

3. **硬件现象判据**
   - 正确：显示节奏稳定、无明显闪烁、数字顺序正确。
   - 若显示全灭/全亮：优先检查 `seg/an` 极性是否反了（低有效/高有效）。

### 风险与应对
- 风险1：Basys3 七段极性与预设不一致。
  - 应对：快速翻转段码与位选极性进行 A/B 验证。
- 风险2：程序延时过短导致肉眼看不清。
  - 应对：增大软件延时循环计数。
- 风险3：`tohost` 写频率影响 UART。
  - 应对：显示与 UART 解耦，仅锁存低 4 位，不依赖 UART 状态。
